Frontend Basics

Tips and tools to make your frontend cleaner and stronger.

Why should we care?

  • It's easy to do. You literally just have to care - just a tiny, little, bit.
  • Less code makes for faster development
  • Polyfills make browser support easy
  • Boilerplates standardize your workflow and make everything faster

Mixin frameworks kinda suck

Things like Compass and Neat have some cool tools with them (sprite sheet generation, data URI generation, etc.) but 99% of the time we don't use these frameworks for that functionality. We primarily use them to apply vendor prefixes and use silly mixins that obfuscate our code. For example, the "size" mixin that simply sets width and height at the cost of obfuscating the code.

Autoprefixer

Autoprefixer is one of my favorite tools. It intelligently pulls only the prefixes you need from the CanIUse database rather than applying every prefix (even ones that don't need it) like most frameworks do.

Your code is CSS spec which means it's future-proof and team-proof. By team-proof I mean if everyone specializes in a different framework then there's a lot of context changing going on for every project and sometimes within the same project. It's also not tied to a particular preprocessor.

If you're interested in writing sweet, sweet, CSS spec, then Myth takes Autoprefixer's strategy a step further and tries to cover everything, not just vendor prefixes.

CSS frameworks are amazing

Frameworks that create reusable classes and standardize everything. Anyone can grab a copy of Bootstrap and know exactly what to do.

It wouldn't be a bad idea to standardize our workflow based around an existing framework that we simply retheme.

Use these frameworks right

Markup is important with frameworks. Pay attention. I can't count the number of times I've seen Bootstrap grid classes misused.

Creating our own framework

We make a lot of unique sites and apps. We use a lot of interesting tools. We should build our own framework based around our style.

Parts to a boilerplate

  • Reset and clearfix
  • HTML5 Boilerplate
  • Conditionally loaded polyfills

Great tools for great sites

  • Autoprefixer
  • Retina mixin
  • Picturefill
  • Grid system

Develop faster

  • Preprocessors
  • LiveReload
  • Chrome Workspaces

Things to consider

  • Browser support - IE8 is the most popular version of IE. Ignore that market at your own peril.
  • Use JS to style things that are sillily hard with just CSS. jQuery 1.x.x supports IE9 and below pretty seamlessly. Things like cross-browser dynamic-height sticky footers, and vertical centering, are made trivial with a bit of JS
  • Test responsively as you develop

How to test in IE

BrowserStack is slow. VMs - VirtualBox and ievms (make sure you have a lot of space). You can navigate to localhost with http://10.0.2.2

Again, test as you develop and it's much easier. Going back and fixing a site that is fundamentally broken is always painful, but fixing a small issue when it pops up is easy. I suggest you just have a couple of these VMs open at all times.

Talk it out

Frontend code is important. It turns into a gross mess of spaghetti if you let it and if things aren't used correctly in the first place then it's really hard to fix it. From time to time, harass other frontend devs just so they can put eyes on it. Don't wait until you get to the PR stage to get shot down and you'll all develop a love for clean frontend code.
